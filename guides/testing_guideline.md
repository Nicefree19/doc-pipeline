# 자동 품질 검사 및 테스트 지침 (Testing Guideline)

에이전트는 코드 작성을 마친 직후, 스스로를 의심하고 테스트를 거쳐야 합니다. 이 문서는 작성된 기능의 신뢰성을 담보하기 위한 기준입니다.

## 1. 테스트 작성 시기
- **기존 기능 수정 시:** 기존 테스트가 깨지는지(Regression) 확인할 것. 필수로 `pytest -q` 등을 수행.
- **신규 기능 추가 시:** 정상 동작 흐름(Happy Path)과 잘못된 입력에 대한 방어 로직(Edge Cases/Error Path)을 반드시 함께 테스트하는 유닛 테스트(Unit Test)를 작성합니다.

## 2. Lint와 Type Checking (품질 게이트 강제)
- 코드 작업 후 `ruff` 등을 통한 린트(Lint) 검사와 `mypy` 등을 사용한 정적 타입 체크가 통과되는 상태를 유지합니다.
- 타입 어노테이션(Type Hints)을 명확히 명시하고 강제합니다 (`Any` 오남용 자제).

## 3. Mocking 전략
- 단위 테스트 시 외부 시스템 의존성(ex. `Path.home()`, `db.cursor()`, 웹훅 요청)은 반드시 목업(Mock) 처리하여, 테스트 환경에서 부수 효과(Side effect)나 네트워크 대기가 발생하지 않게 설정합니다.
